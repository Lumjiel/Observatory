## 一、垃圾回收核心概念

垃圾回收（Garbage Collection，简称GC），本质是JVM（Java虚拟机）提供的“自动清洁工”——它会在后台默默监控内存，把“没人用的对象”（垃圾）占用的内存释放掉，还给JVM重复利用。

我们写Java代码时，不用像C/C++那样手动调用free()释放内存，全靠GC帮我们搞定，这就大大减少了两个致命问题：

- 内存泄漏：没用的对象占着内存不放，久而久之内存被耗尽；
    
- 内存溢出：想创建新对象，但内存不够用（GC没及时回收，或回收后也没有足够空间）。
    

举个生活化例子：你家里的垃圾桶（堆内存），你用完的废纸、空瓶子（没用的对象）会扔进去，GC就像清洁工，定期来把垃圾桶清空，腾出空间放新的垃圾（新对象）。

### 1.1 什么时候会触发GC？

GC不是“随时都在扫”，也不是“想扫就扫”，它的触发有明确条件，最常见的有4种：

#### （1）堆内存不足

当JVM尝试给新对象分配内存，但堆内存已经满了，没地方放新对象时，会**强制触发GC**，尝试释放垃圾对象的内存，再给新对象分配空间。

实例：我们可以通过JVM参数控制堆内存大小，模拟这种场景：

```java
// 启动参数：-Xms5m -Xmx5m（初始堆5M，最大堆5M，堆内存固定5M）
public class GCTest {
    public static void main(String[] args) {
        // 循环创建对象，直到堆内存满，触发GC
        for (int i = 0; ; i++) {
            // 每次创建1M大小的对象（1个byte[]占1M）
            byte[] arr = new byte[1024 * 1024]; 
        }
    }
}
```

运行后，JVM会频繁触发GC，最后还是会因为内存不足，抛出OutOfMemoryError（内存溢出异常）——因为创建的对象太多，GC回收速度赶不上对象创建速度。

#### （2）手动请求GC

开发者可以在代码中，通过两个方法“建议”JVM执行GC，但JVM有权忽略这个建议（就像你提醒清洁工该打扫了，但清洁工可能手头有更急的事，暂时不扫）。

两种手动请求方式（效果完全一样）：

```java
// 方式1：System类的静态方法
System.gc();
// 方式2：Runtime类的实例方法（Runtime是单例，通过getRuntime()获取实例）
Runtime.getRuntime().gc();
```

注意：不要依赖手动GC！比如你想在创建大量临时对象后，手动触发GC释放内存，但JVM可能不会立即执行，反而会影响程序性能（频繁手动请求，可能打乱GC的优化策略）。

#### （3）JVM参数配置触发

启动Java程序时，我们可以通过JVM参数，强制GC按照指定规则触发，常见参数如下（重点记前2个）：

- -Xms：初始堆内存（比如-Xms10m，程序启动时就分配10M堆内存）；
    
- -Xmx：最大堆内存（比如-Xmx20m，堆内存最多不能超过20M）；
    
- -XX:+PrintGCDetails：打印GC详细日志（能看到GC触发时间、回收内存大小、STW时间等，调试常用）；
    
- -XX:MaxTenuringThreshold：设置对象晋升老年代的年龄阈值（默认15，后面会详细说）。
    

实例：通过-Xmx设置小堆内存，快速触发GC，同时打印GC日志：

启动命令：java -Xms5m -Xmx5m -XX:+PrintGCDetails GCTest

运行后，控制台会输出GC日志，能清晰看到“什么时候触发GC、回收了多少内存”。

#### （4）对象数量/内存使用达到阈值

JVM的垃圾收集器内部，会监控“对象创建数量”“内存使用率”等指标，当这些指标达到预设阈值时，会自动触发GC。

比如：新生代的Eden区（后面会讲），默认使用率达到100%（满了）就触发Minor GC；老年代内存使用率达到某个比例（比如90%，可通过参数配置），就触发Major GC。

### 1.2 核心前提：怎么判断一个对象是“垃圾”？

GC要回收垃圾，首先得明确：**哪个对象是“没人用”的**？——也就是“不可达对象”（没有任何变量引用它，程序后续再也无法使用它）。

JVM判断垃圾的核心有2种算法，其中“可达性分析算法”是主流（解决了另一种算法的致命缺陷）。

## 二、垃圾判断算法

### 2.1 引用计数法（淘汰算法，了解即可）

这是最朴素、最简单的算法，核心思想：给每个对象贴一个“引用计数器”，记录当前引用它的变量数量。

#### 原理（3步）：

1. 对象被创建时，计数器初始化为1（比如User user = new User()，user引用该对象，计数器=1）；
    
2. 有新的变量引用该对象时，计数器+1（比如User user2 = user，计数器=2）；
    
3. 某个引用被释放时（比如user = null），计数器-1；当计数器减为0时，说明对象没人用了，标记为垃圾，等待回收。
    

#### 实例演示：

```java
public class ReferenceCountTest {
    public static void main(String[] args) {
        // 1. 创建对象A，计数器=1
        User a = new User("A");
        // 2. 创建对象B，计数器=1
        User b = new User("B");
        
        // 3. a引用B，B的计数器+1 → B计数器=2
        a.setFriend(b);
        // 4. b引用A，A的计数器+1 → A计数器=2
        
        // 5. 释放a和b的引用，A和B的计数器各减1 → 均为1
        a = null;
        b = null;
        
        // 此时A和B计数器都是1，但没有任何外部引用，属于垃圾，但引用计数法无法识别
        System.gc();
    }
}

class User {
    String name;
    User friend; // 引用其他User对象
    
    public User(String name) {
        this.name = name;
    }
    
    public void setFriend(User friend) {
        this.friend = friend;
    }
}
```

#### 致命缺点：无法解决“循环引用”问题

就像上面的实例：A引用B，B引用A，两者的计数器都是1，但没有任何外部变量引用它们（a和b都被置为null），程序后续再也无法使用A和B——它们明明是垃圾，但引用计数法因为计数器没减到0，无法识别，也就无法回收。

这就是为什么JVM不使用引用计数法，而是用“可达性分析算法”。

### 2.2 可达性分析算法（JVM主流算法，重点掌握）

核心思想：**从“根对象”（GC Roots）出发，像“查家谱”一样，向下追溯所有被引用的对象；如果某个对象，无法通过任何一条引用链，追溯到GC Roots，说明它是不可达的，标记为垃圾**。

这个算法完美解决了“循环引用”问题——比如上面的A和B，虽然互相引用，但它们无法追溯到GC Roots，所以会被标记为垃圾，正常回收。

#### 第一步：明确“GC Roots（根对象）”有哪些？

GC Roots是JVM认定的“绝对不会被回收”的对象（相当于“家谱的老祖宗”），只有4类，每类都有明确的场景，结合实例理解：

##### （1）虚拟机栈中正在引用的对象（最常见）

虚拟机栈（栈内存）中，每个方法执行时都会创建一个“栈帧”，栈帧里的局部变量引用的对象，就是GC Roots。

原因：局部变量和方法“绑定”，只要方法没执行完，局部变量就一直在用，JVM必须保证它不被回收（否则会出现空指针异常）。

实例：

```java
public static void main(String[] args) {
    // user是main方法栈帧中的局部变量，引用User对象
    User user = new User("张三");
    // 只要main方法没执行完，user引用的对象就是GC Roots，不会被回收
    doSomething(user);
}

public static void doSomething(User u) {
    // u是doSomething方法栈帧中的局部变量，引用同一个User对象
    // 只要doSomething没执行完，该对象依然是GC Roots
    System.out.println(u.name);
}
```

##### （2）本地方法栈中正在引用的对象

本地方法栈和虚拟机栈类似，区别是：虚拟机栈执行的是Java方法，本地方法栈执行的是“native方法”（由C/C++编写，比如System.currentTimeMillis()底层就是native方法）。

原因：JVM无法感知native方法的执行状态，为了保证native方法能正常运行，它引用的对象必须被当作GC Roots，不能被回收。

##### （3）方法区中静态属性引用的对象

方法区（元空间，Metaspace）中，存储类的元数据（比如类的结构、静态变量、常量等），其中静态变量引用的对象，是GC Roots。

原因：静态变量和“类”绑定，不是和“对象”绑定——类被JVM加载后，它的元数据会一直存在于方法区，直到类被卸载（类卸载的条件极其苛刻，比如没有任何对象引用该类、类加载器被回收，核心库的类几乎永远不会被卸载）。

实例：

```java
class Student {
    // 静态变量（和Student类绑定），引用User对象
    public static User teacher = new User("李老师");
}

public class GCRootsTest {
    public static void main(String[] args) {
        // 即使没有创建Student对象，静态变量teacher依然存在（和类绑定）
        // teacher引用的User对象，是GC Roots，不会被回收
    }
}
```

##### （4）方法区中常量引用的对象

方法区的“运行时常量池”中，常量引用的对象，是GC Roots。

原因：常量是全局共享的，由JVM直接管理，只要常量没被移除（一般不会移除），它引用的对象就不会被回收。

实例：

```java
class ConstantTest {
    // 常量（运行时常量池中的常量），引用User对象
    public static final User CONST_USER = new User("常量对象");
}
```

CONST_USER是常量，一旦被创建，就会一直存在于运行时常量池，它引用的User对象，永远是GC Roots，不会被回收。

#### 第二步：可达性分析的执行流程（通俗版）

1. JVM先找到所有GC Roots对象（相当于“老祖宗”）；
    
2. 从每个GC Roots出发，向下遍历所有被它引用的对象（相当于“儿子”），再遍历这些对象引用的对象（相当于“孙子”），以此类推；
    
3. 所有能被遍历到的对象，都是“可达对象”（有用的对象），不会被回收；
    
4. 无法被遍历到的对象，都是“不可达对象”（垃圾），等待被回收。
    

#### 补充：Java中的4种引用类型（关联垃圾判断）

很多人会有疑问：“如果一个对象被引用了，就一定不会被回收吗？”——不一定！Java中的引用分为4种，不同引用类型，对象的“存活优先级”不同，这也会影响GC的回收策略（原文提到“新开文章写”，这里补充核心内容，方便理解）。

|引用类型|核心特点|实例|GC回收策略|
|---|---|---|---|
|强引用（默认）|最常用，对象被强引用关联时，绝对不会被回收|User user = new User()|只有当强引用被释放（user=null），对象才可能被回收|
|软引用|用于缓存（比如图片缓存），内存充足时不回收，内存不足时回收|SoftReference<User> soft = new SoftReference<>(new User())|堆内存不足时，会优先回收软引用关联的对象|
|弱引用|生命周期更短，无论内存是否充足，只要GC触发，就会回收|WeakReference<User> weak = new WeakReference<>(new User())|GC一旦执行，就会回收弱引用关联的对象（即使内存充足）|
|虚引用|“幽灵引用”，无法通过虚引用获取对象，仅用于监听对象回收|PhantomReference<User> phantom = new PhantomReference<>(new User(), queue)|对象被回收前，会将虚引用加入队列，通知程序|

重点：我们平时写代码，用的都是“强引用”（默认），这也是为什么“对象被引用就不会被回收”的原因；而软引用、弱引用，主要用于一些特殊场景（比如缓存），减少内存占用。

## 三、垃圾回收算法（3种核心，附图解思路+适用场景）

判断出垃圾对象后，GC就要“清理垃圾”——释放垃圾对象占用的内存。这时候就需要“垃圾回收算法”，JVM核心有3种算法，各有优缺点，适用于不同场景（结合分代回收使用）。

### 3.1 标记-清除算法（Mark-Sweep）

最基础、最原始的算法，核心是“先标记，再清除”，两步完成垃圾回收，就像“先在垃圾上画叉，再把画叉的垃圾扔掉”。

#### 核心流程（2步）：

1. 标记阶段：以GC Roots为起点，通过可达性分析，给所有“存活对象”打上“存活标记”（比如给对象的头信息加一个标识位，1表示存活，0表示死亡）；
    
2. 清除阶段：遍历整个堆内存，找到所有“未标记的死亡对象”，释放它们占用的内存；同时，将释放后的空闲内存块，记录到“空闲内存链表”中（方便后续给新对象分配内存时，快速找到合适的空闲块）。
    

#### 关键疑问解答（原文补充，更易懂）：

##### 疑问1：为什么标记“存活对象”，而不是直接标记“死亡对象”？

效率更高！Java对象有个核心特性——“朝生夕死”：大部分对象都是临时对象，生命周期极短（比如循环中创建的临时对象、方法中创建的局部对象），存活对象的数量，远少于死亡对象的数量。

比如：程序运行时，创建了1000个对象，其中990个都是临时对象（已经死亡），只有10个存活。如果标记存活对象，只需要遍历10个；如果标记死亡对象，需要遍历990个——明显标记存活对象更高效。

尤其在新生代（后面会讲），这种“存活少、死亡多”的特点更明显，标记存活对象的优势更大。

##### 疑问2：为什么清除后，要组织“空闲内存链表”？

避免内存碎片浪费！清除死亡对象后，释放的内存块大小不一、位置分散（比如A对象释放1M，B对象释放2M，两者之间隔着一个存活对象），这些分散的小内存块，就是“内存碎片”。

如果不组织空闲链表，后续创建一个3M的大对象，虽然总空闲内存有3M（1M+2M），但因为是分散的碎片，无法容纳大对象，导致内存浪费；而将空闲内存块组织成链表，记录每个块的大小和起始位置，后续分配内存时，就可以遍历链表，找到“大小合适”的空闲块，充分利用内存碎片。

#### 优缺点总结（重点）：

|优点|缺点|
|---|---|
|1. 简单高效，实现难度低；2. 没有对象移动操作，适合大对象较多的场景（大对象移动成本高）；3. 无内存拷贝开销，清除后直接回收内存。|1. 产生大量内存碎片，导致内存分配效率低；2. 可能无法满足大对象的内存分配需求（碎片总大小足够，但单个碎片太小）；3. 清除阶段需要遍历整个堆内存，效率较低（当堆内存较大时）。|

#### 图解思路（文字描述，方便理解）：

初始堆内存：[存活对象A][死亡对象B][存活对象C][死亡对象D][存活对象E]
标记阶段后：[存活A（标记）][死亡B（未标记）][存活C（标记）][死亡D（未标记）][存活E（标记）] 
清除阶段后：[存活A][空闲块1（B的内存）][存活C][空闲块2（D的内存）][存活E] 
空闲链表：空闲块1（大小=B的内存）→ 空闲块2（大小=D的内存））

### 3.2 标记-整理算法（Mark-Compact）

为了解决“标记-清除算法的内存碎片问题”而诞生，核心是“标记后，先整理存活对象，再清除垃圾”——相当于“先把有用的东西挪到一起，再把剩下的垃圾全部扔掉”。

#### 核心流程（3步）：

1. 标记阶段：和“标记-清除算法”完全一样，给所有存活对象打上存活标记；
    
2. 整理阶段：将所有存活对象，向堆内存的“一端”（比如起始地址）紧凑排列，消除存活对象之间的空闲内存（相当于“搬家”）；
    
3. 清除阶段：直接清理“存活对象边界外”的所有内存（此时边界外的内存都是连续的，没有碎片）。
    

#### 关键注意点：

存活对象“搬家”后，它的内存地址会发生变化——这就需要修改“所有引用该对象的引用地址”（比如A对象原来的地址是0x123，搬家后变成0x456，所有引用A的变量，都要改成0x456），否则会出现空指针异常。

#### 优缺点总结（重点）：

|优点|缺点|
|---|---|
|1. 没有内存碎片，内存分配效率高（大对象可以轻松找到连续的空闲内存）；2. 内存利用率高，不会浪费碎片空间。|1. 增加了“对象移动”和“引用地址更新”的开销，消耗大量CPU资源；2. STW（Stop The World）时间更长（移动对象、更新引用都需要暂停用户线程），对时间敏感的系统（比如电商支付）不友好。|

#### 图解思路（文字描述）：

标记后：[存活A（标记）][死亡B（未标记）][存活C（标记）][死亡D（未标记）][存活E（标记）] 
整理后：[存活A][存活C][存活E][空闲块（连续，大小=B+D的内存）] 
清除后：[存活A][存活C][存活E][空闲块（连续）]

### 3.3 复制算法（Copying）

核心思想：“空间换时间”——将堆内存划分为两个大小相等的区域，每次只使用其中一个区域；回收时，将存活对象复制到另一个空闲区域，再清空当前使用的区域。

因为Java对象“朝生夕死”，存活对象少，复制的开销很小，所以效率极高。

#### 核心流程（3步）：

1. 内存划分：将堆内存分为两个大小相等的区域，比如“From区”和“To区”（或“S0区”和“S1区”），初始时只使用From区，To区空闲；
    
2. 标记复制：当From区满了，触发GC，标记From区中的所有存活对象，将它们全部复制到To区（因为存活对象少，复制速度快）；
    
3. 清空切换：清空From区的所有内存，然后交换From区和To区的角色（原来的To区变成新的From区，原来的From区变成新的To区），下次分配对象时，使用新的From区。
    

#### 优缺点总结（重点）：

|优点|缺点|
|---|---|
|1. 没有内存碎片，内存分配效率极高（直接在连续的空闲区域分配）；2. 实现简单，回收速度快（只需复制存活对象，再清空当前区）；3. 不需要标记死亡对象，只需标记存活对象并复制。|1. 内存浪费严重（只能使用一半的堆内存，另一半永远空闲，用于复制）；2. 有对象复制开销（如果存活对象多，复制速度会变慢）。|

#### 图解思路（文字描述）：

初始状态：From区（使用中）→ 对象1对象2对象3对象4，To区（空闲）→ []
GC触发（From区满）：标记存活对象（假设对象1、3存活），复制到To区 → To区：对象1对象3]
清空切换：清空From区，交换角色 → 新From区（原To区）：对象1对象3，新To区（原From区）：[]

### 3.4 三种算法对比（快速选型参考）

|算法|核心优势|核心劣势|适用场景|
|---|---|---|---|
|标记-清除|无对象移动，适合大对象|内存碎片多|老年代（大对象多、存活对象多）|
|标记-整理|无内存碎片|STW时间长、CPU开销大|老年代（内存碎片敏感场景）|
|复制|效率高、无碎片|内存浪费一半|新生代（存活对象少、临时对象多）|

## 四、JVM分代回收机制（重点，结合算法实际应用）

上面讲的3种算法，JVM并不是“单独使用某一种”，而是根据“对象的生命周期不同”，将堆内存分为“新生代”和“老年代”，分别使用合适的算法——这就是“分代回收机制”，也是JVM GC的核心设计。

核心依据：Java对象的生命周期差异极大，可分为两类：

- 临时对象：生命周期极短（毫秒级、秒级），创建后很快变成垃圾（比如循环中的临时对象、方法中的局部对象），占对象总数的90%以上；
    
- 长期对象：生命周期很长（分钟级、小时级，甚至和程序生命周期一致），比如静态变量引用的对象、缓存对象。
    

因此，JVM将堆内存分为“新生代”（存放临时对象）和“老年代”（存放长期对象），分别采用不同的回收算法和策略，提升GC效率。

### 4.1 堆内存分代结构（重点记比例）

堆内存 = 新生代 + 老年代（默认比例：新生代占1/3，老年代占2/3）；

新生代 = Eden区 + 2个Survivor区（From区、To区），默认比例：Eden区占80%，From区占10%，To区占10%；

注意：

- 新生代的2个Survivor区，**永远有一个是空闲的**（用于复制算法的“目标区域”）；
    
- 新生代实际可使用的内存 = Eden区 + From区（共90%），To区（10%）空闲，用于复制存活对象；
    
- 可以通过JVM参数调整比例，比如-XX:SurvivorRatio=8（设置Eden区和Survivor区的比例为8:1）。
    

### 4.2 新生代回收（Minor GC / Young GC）

新生代专门存放“临时对象”，特点是“存活对象少、死亡对象多”，因此采用**复制算法**，回收效率极高，STW时间短。

#### 核心流程（结合复制算法，附实例）：

1. 对象创建：新对象优先分配到Eden区（80%的新生代内存），如果Eden区满了，就尝试分配到From区；
    
2. 触发GC：当Eden区满了，无法分配新对象时，触发Minor GC（新生代回收）；
    
3. 标记复制：标记Eden区和From区中的所有存活对象，将它们全部复制到To区；
    
4. 对象晋升：存活对象在Survivor区（From/To）中，每经历一次Minor GC，“年龄”就加1（年龄记录在对象头中）；当年龄达到阈值（默认15，可通过-XX:MaxTenuringThreshold调整），该对象会被“晋升”到老年代；
    
5. 清空切换：清空Eden区和From区，然后交换From区和To区的角色（原来的To区变成新的From区，原来的From区变成新的To区），下次Minor GC时，使用新的From区。
    

#### 特殊情况：分配担保策略

如果Minor GC时，To区的空间不足，无法容纳所有存活对象，此时会触发“分配担保”——将存活对象直接晋升到老年代（相当于“To区装不下，就把多余的对象送到老年代”）。

#### Minor GC的特点：

- 触发频率高（因为临时对象多，Eden区很快就会满）；
    
- STW时间短（复制算法效率高，存活对象少，复制速度快）；
    
- 回收范围：Eden区 + 一个Survivor区（From区）；
    
- 核心算法：复制算法。
    

### 4.3 老年代回收（Major GC / Old GC）

老年代专门存放“长期对象”（从新生代晋升来的对象、直接分配的大对象），特点是“存活对象多、死亡对象少”，因此采用**标记-清除算法**或**标记-整理算法**（避免复制大量存活对象，节省CPU开销）。

#### 触发场景（3种核心）：

1. 老年代内存不足：当老年代的内存使用率达到预设阈值（比如90%），或者无法给新晋升的对象分配内存时，触发Major GC；
    
2. Minor GC时分配担保失败：Minor GC时，To区装不下存活对象，需要晋升到老年代，但老年代也没有足够空间，触发Major GC；
    
3. 主动调用System.gc()：JVM可能会响应这个请求，触发Major GC（但不保证）。
    

#### Major GC的特点：

- 触发频率低（老年代对象生命周期长，不容易满）；
    
- STW时间长（比Minor GC长一个数量级，可能是几十毫秒，甚至上百毫秒）；
    
- 回收范围：主要是老年代（不同JVM实现可能略有差异）；
    
- 核心算法：标记-清除算法（默认，效率高）或标记-整理算法（内存碎片敏感场景）。
    

### 4.4 全量回收（Full GC）

Full GC是“最重量级”的GC，会回收**新生代 + 老年代**（可能还包括元空间），STW时间最长（秒级），对应用性能影响极大，应尽量避免。

#### 触发场景（5种核心，重点记）：

1. 老年代满，且Major GC无法释放足够空间（最常见）；
    
2. 元空间（Metaspace）满：元空间存放类的元数据，当加载的类太多（比如频繁动态生成类），元空间满了，触发Full GC；
    
3. Minor GC时分配担保失败：Survivor区装不下存活对象，老年代也没有足够空间接收晋升对象；
    
4. 主动调用System.gc()：JVM可能会触发Full GC（但不保证）；
    
5. JVM参数配置触发：比如设置了-XX:+ExplicitGCInvokesConcurrent（手动调用System.gc()时触发Full GC）。
    

#### Full GC的特点：

- 回收范围最广：新生代 + 老年代 + （可选）元空间；
    
- STW时间最长：秒级（甚至更长），会导致应用卡顿、响应变慢；
    
- 核心算法：新生代用复制算法，老年代用标记-清除/标记-整理算法；
    
- 优化方向：尽量减少Full GC的触发次数（比如合理设置堆内存大小、避免频繁加载大量类）。
    

### 4.5 分代回收整体流程（通俗总结）

1. 新对象创建，优先分配到Eden区；
    
2. Eden区满，触发Minor GC，回收Eden区和From区的垃圾，存活对象复制到To区，年龄+1；
    
3. 存活对象在Survivor区反复经历Minor GC，年龄达到15，晋升到老年代；
    
4. 老年代内存不足，触发Major GC，回收老年代垃圾；
    
5. 如果Major GC无法释放足够空间，或其他触发条件满足，触发Full GC，回收整个堆内存；
    
6. Full GC后依然没有足够空间，抛出OutOfMemoryError（内存溢出）。
    

## 五、垃圾回收器（GC执行的“具体工具”）

前面讲的“垃圾判断算法”“垃圾回收算法”“分代回收机制”，都是“理论”；而垃圾回收器（Garbage Collector），是JVM中“实际执行GC操作的工具”——不同的垃圾回收器，实现了不同的回收策略，适用于不同的应用场景。

JVM内置了多种垃圾回收器，核心分为4类（按“回收策略”划分），我们重点掌握常用的5种：Serial/Serial Old、Parallel Scavenge/Parallel Old、ParNew/CMS、G1。

### 先明确2个关键概念（避免混淆）：

#### 并行（Parallel）：

多个GC线程“同时执行”（利用多核CPU，每个核心一个GC线程），但**用户线程会被暂停**（STW）；比如：3个GC线程同时回收垃圾，用户线程全部暂停，直到GC完成。

通俗例子：家里有3个灶台（多核CPU），3个厨师（GC线程）同时做饭（回收垃圾），你（用户线程）只能等着，直到饭做好（GC完成）才能吃饭。

#### 并发（Concurrent）：

GC线程和用户线程“交替执行”（CPU在两者之间快速切换），**用户线程不会被长时间暂停**；比如：GC线程执行10ms，用户线程执行10ms，反复交替，用户几乎感觉不到卡顿。

通俗例子：家里只有1个灶台（单核CPU），厨师（GC线程）做10分钟饭，你（用户线程）用10分钟吃饭，反复交替，你不需要一直等着。

关键区别：并行是“GC线程多，用户线程暂停”，追求“高吞吐量”；并发是“GC线程和用户线程交替，用户线程少暂停”，追求“低延迟”。

### 5.1 串行回收器（Serial + Serial Old）—— 单线程回收

最基础、最简单的回收器，**单线程执行GC**（只有1个GC线程），回收时，所有用户线程都会被暂停（STW）。

#### 组成（分代对应）：

- Serial收集器：用于新生代，采用复制算法；
    
- Serial Old收集器：用于老年代，采用标记-整理算法。
    

#### 核心特点：

- 单线程GC，实现简单，内存开销小（不需要维护多个GC线程的通信）；
    
- STW时间较长（只有1个GC线程，回收速度慢）；
    
- 不依赖多核CPU，单核CPU也能高效运行。
    

#### 适用场景：

客户端应用（比如桌面程序、 Swing程序）、单核CPU环境、小内存堆（<1GB）—— 这类应用对“延迟”要求不高，STW时间长一点也能接受。

#### 启动参数：

-XX:+UseSerialGC（强制使用Serial + Serial Old回收器）

### 5.2 吞吐量优先回收器（Parallel Scavenge + Parallel Old）—— 并行回收

核心目标：**追求高吞吐量**（吞吐量 = 用户线程运行时间 / （用户线程运行时间 + GC线程运行时间）），适用于“后台服务、数据处理”等对吞吐量敏感，对延迟不敏感的场景。

#### 组成（分代对应）：

- Parallel Scavenge收集器：用于新生代，采用复制算法，并行GC线程；
    
- Parallel Old收集器：用于老年代，采用标记-整理算法，并行GC线程。
    

#### 核心特点：

- 并行GC：多个GC线程同时执行，回收速度比Serial快，STW时间比Serial短；
    
- 自适应调节（核心优势）：开启-XX:+UseAdaptiveSizePolicy后，JVM会自动优化堆内存大小、Survivor区比例、GC触发频率等参数，不需要开发者手动调参；
    
- 可手动设置吞吐量目标：通过-XX:GCTimeRatio=ratio（设置GC时间占比）、-XX:MaxGCPauseMillis=ms（设置最大STW时间）。
    

#### 适用场景：

后端服务、数据处理、大数据任务、吞吐量优先的服务器应用（比如Tomcat后台接口、数据ETL程序）—— 这类应用需要高效利用CPU，尽可能多的时间用于处理业务逻辑（用户线程）。

#### 启动参数：

-XX:+UseParallelGC（强制使用Parallel Scavenge + Serial Old，不推荐）

-XX:+UseParallelOldGC（强制使用Parallel Scavenge + Parallel Old，推荐）

### 5.3 响应时间优先回收器（ParNew + CMS）—— 并发回收

核心目标：**追求低延迟**（尽可能缩短STW时间），适用于“对延迟敏感的应用”（比如电商支付、秒杀系统、接口响应时间要求<100ms的场景）—— 这类应用需要用户线程尽可能少暂停，避免卡顿。

#### 组成（分代对应）：

- ParNew收集器：用于新生代，采用复制算法，并行GC线程（Serial的多线程版本）；
    
- CMS收集器（Concurrent Mark Sweep）：用于老年代，采用标记-清除算法，并发GC线程（核心）。
    

#### 核心特点：

- 并发回收（核心）：CMS的核心阶段（并发标记、并发清除）和用户线程交替执行，STW时间极短（毫秒级）；
    
- 新生代并行回收：ParNew采用多线程复制，STW时间短；
    
- 缺点明显：内存碎片多（老年代用标记-清除算法）、对CPU资源敏感、存在并发失败和浮动垃圾问题。
    

#### CMS核心流程（4步，重点记STW阶段）：

1. 初始标记（STW，快速）：仅标记GC Roots直接引用的对象（第一级对象），因为对象数量少，STW时间很短（几毫秒）；
    
2. 并发标记（并发，无STW）：GC线程和用户线程交替执行，从初始标记的对象出发，遍历整个引用链，标记所有存活对象；—— 这一步耗时最长，但用户线程不暂停；
    
3. 重新标记（STW，快速）：修正“并发标记阶段”因用户线程操作导致的标记变动（比如并发标记时，用户线程释放了某个对象的引用，导致该对象变成垃圾），STW时间较短（几毫秒）；
    
4. 并发清除（并发，无STW）：GC线程和用户线程交替执行，清除未标记的死亡对象，释放内存；—— 用户线程不暂停。
    

总结：CMS的STW时间，只集中在“初始标记”和“重新标记”两个阶段，总时长很短，用户几乎无感知。

#### CMS存在的4个核心问题（重点掌握）：

虽然CMS能实现低延迟，但它的设计缺陷也很明显，这也是后续被G1回收器替代的核心原因，每个问题都结合实例和底层逻辑讲解，方便理解和记忆。

##### （1）内存碎片多（最突出问题）

CMS的老年代采用「标记-清除算法」，如前文所述，这种算法只清除死亡对象，不整理存活对象，会产生大量分散的内存碎片。

实例：老年代内存中，存活对象A（占用1M）、存活对象B（占用2M）之间，有空闲碎片0.5M，存活对象B之后有空闲碎片1M，总空闲内存1.5M；此时程序需要创建一个1.2M的大对象，虽然总空闲内存足够，但没有一个连续的碎片能容纳该对象，导致无法分配内存，进而触发Full GC（即使总空闲内存充足）。

影响：频繁的内存碎片会导致“内存明明有剩余，却无法分配大对象”，被迫触发Full GC，反而打破了CMS“低延迟”的优势，导致应用卡顿。

补充解决方案：JVM提供了一个参数 `-XX:+UseCMSCompactAtFullCollection`，开启后，每次CMS执行完Full GC，会额外执行一次“内存整理”操作，清除碎片；但这会增加STW时间，相当于“用部分延迟换取内存碎片减少”，是一种折中方案。

##### （2）对CPU资源非常敏感

CMS是「并发回收」，GC线程和用户线程交替执行，这就需要CPU在两者之间分配资源——GC线程会占用一定的CPU核心，导致用户线程可用的CPU资源减少，影响应用的业务处理效率。

底层原因：并发阶段（并发标记、并发清除），GC线程需要和用户线程“抢CPU”，如果CPU核心较少（比如2核、4核），GC线程占用1-2个核心后，用户线程的执行效率会明显下降；如果CPU核心充足（比如16核以上），这种影响会被弱化。

实例：一个电商秒杀系统，部署在4核CPU的服务器上，开启CMS回收器后，并发标记阶段，2个GC线程占用2个CPU核心，剩余2个核心供用户线程处理秒杀请求，导致请求响应时间从50ms延长到150ms，甚至出现请求超时。

补充：CMS的GC线程数量可以通过参数 `-XX:ParallelCMSThreads` 配置，默认值是「CPU核心数 + 3」/ 4，比如4核CPU，默认GC线程数是(4+3)/4=1（取整数），可根据实际CPU配置调整，减少对用户线程的影响。

##### （3）并发失败（Concurrent Mode Failure，CMF）

这是CMS最致命的问题，核心原因：**并发回收阶段（GC线程和用户线程交替执行），用户线程还在持续创建新对象，老年代的内存被快速占用，导致CMS还没回收完垃圾，老年代就满了**。

触发过程（通俗版）：

1. 老年代内存使用率达到阈值（默认92%，可通过 `-XX:CMSInitiatingOccupancyFraction` 配置），触发CMS并发回收；
    
2. 并发标记、并发清除阶段，用户线程继续创建对象，不断占用老年代内存；
    
3. CMS还没完成回收，老年代内存就被占满，此时JVM会立即停止CMS回收，强制触发Full GC；
    
4. Full GC会暂停所有用户线程（STW），采用「标记-整理算法」回收整个堆内存，STW时间会非常长（秒级），导致应用彻底卡顿。
    

实例：老年代内存10G，阈值92%（即9.2G时触发CMS），CMS开始并发回收后，用户线程每分钟创建0.5G对象，而CMS并发回收每分钟只能回收0.3G垃圾，10分钟后，老年代内存被占满，触发Full GC，应用卡顿10秒以上。

解决方案（3个常用方案，结合使用）：

- 降低触发阈值：通过 `-XX:CMSInitiatingOccupancyFraction=80`（比如设置为80%），让CMS更早开始回收，预留足够的内存空间供用户线程创建对象；
    
- 增大老年代内存：通过 `-Xmx` 增大堆内存，同时调整老年代比例，给老年代分配更多内存，减少“回收速度赶不上对象创建速度”的情况；
    
- 开启内存整理：结合 `-XX:+UseCMSCompactAtFullCollection`，减少内存碎片，避免因碎片导致的“虚假内存不足”，间接减少并发失败。
    

##### （4）浮动垃圾（Floating Garbage）

定义：**在CMS的并发标记阶段，GC线程正在标记存活对象，此时用户线程修改了对象的引用关系，导致部分原本“可达”的对象变成“不可达”（垃圾），但这些垃圾没有被标记，无法在本次GC中回收，只能等到下一次GC才能回收**——这部分未被标记的垃圾，就是浮动垃圾。

底层逻辑：并发标记阶段，GC线程和用户线程同时执行，比如：GC线程刚标记完对象C是存活对象（因为对象B引用了C），此时用户线程执行 `b = null`（释放对象B对C的引用），对象C变成不可达垃圾，但GC线程已经完成了对C的标记，不会再重新标记，导致C无法在本次CMS回收中被清除，只能成为浮动垃圾。

影响：浮动垃圾会占用老年代内存，减少可用内存空间，增加“并发失败”的风险——如果浮动垃圾过多，会导致老年代内存快速被占满，进而触发CMF，引发Full GC。

补充：浮动垃圾是并发回收的“固有缺陷”（只要GC线程和用户线程并发执行，就可能出现），无法彻底解决，只能通过“更早触发CMS回收”“预留足够内存”等方式，减少其影响。

### 5.4 G1回收器（Garbage-First）—— 区域化分代回收（JDK1.7+主流）

G1回收器是JVM为了解决“CMS的缺陷”而设计的，核心定位：**兼顾吞吐量和低延迟**，既不像CMS那样对CPU敏感、有并发失败风险，也不像Parallel那样延迟过高，是目前大多数服务器应用（Java后端、Android服务）的首选回收器。

核心设计亮点：**区域化分代回收**—— 打破了“新生代和老年代物理隔离”的传统分代模式，将堆内存划分为多个大小相等的“独立区域（Region）”，每个Region可以灵活扮演“Eden区、Survivor区、老年代区”，无需固定比例，根据对象的生命周期动态调整。

#### 1. G1的核心结构（Region划分）

堆内存被划分为N个大小相等的Region，每个Region的大小可通过 `-XX:G1HeapRegionSize` 配置（默认是堆内存的1/2000，范围1MB~32MB，必须是2的幂次方），比如堆内存10G，可划分为1024个10MB的Region。

Region的3种类型（动态切换）：

- Eden Region：存放新创建的临时对象（和传统新生代的Eden区功能一致）；
    
- Survivor Region：存放新生代回收后存活的对象（和传统Survivor区一致）；
    
- Old Region：存放长期对象、从新生代晋升来的对象（和传统老年代一致）；
    
- 补充：还有一种“Humongous Region”（巨型区域），专门存放“大对象”—— 当对象大小超过Region大小的50%，就会直接分配到Humongous Region（避免大对象在多个Region之间复制，节省开销）。
    

核心优势：Region的动态切换，让G1可以“按需回收”—— 优先回收“垃圾最多的Region”（这也是Garbage-First的名字由来），比如某个Region中90%都是垃圾，G1会优先回收这个Region，提升回收效率；同时，无需固定新生代和老年代的比例，JVM可根据对象生命周期动态调整，更灵活。

#### 2. G1的核心特点（对比CMS/Parallel）

- 兼顾吞吐量和低延迟：并发回收（减少STW时间）+ 并行回收（提升回收速度），默认追求“低延迟”，也可通过参数调整为“高吞吐量”；
    
- 无内存碎片：老年代回收采用「标记-整理算法」（区别于CMS的标记-清除），回收后会整理存活对象，彻底解决内存碎片问题；
    
- 按需回收：优先回收垃圾多的Region，避免“回收整个新生代/老年代”，减少STW时间；
    
- 支持可预测的STW时间：通过参数 `-XX:MaxGCPauseMillis` 可以设置“最大STW时间”（比如100ms），G1会自动调整回收策略，尽量保证STW时间不超过该值（适合对延迟有明确要求的场景）；
    
- 缺点：内存开销比CMS大（需要维护每个Region的状态、引用关系），实现复杂，在小内存堆（&lt;4GB）中的性能不如Parallel回收器。
    

#### 3. G1的核心回收流程（4步，重点记STW阶段）

G1的回收流程融合了“分代回收”和“区域化回收”的特点，核心分为4个阶段，其中只有2个阶段会触发STW，且STW时间可控制：

##### （1）初始标记（STW，快速）

和CMS的初始标记一致：仅标记GC Roots直接引用的对象（第一级对象），不遍历整个引用链，因为对象数量少，STW时间很短（几毫秒），用户无感知。

##### （2）并发标记（并发，无STW）

从初始标记的对象出发，GC线程和用户线程交替执行，遍历整个引用链，标记所有存活对象；同时，记录用户线程对对象引用关系的修改（通过“写屏障”技术），用于后续修正标记结果。

补充：写屏障（Write Barrier）是G1用来跟踪引用变化的核心技术，相当于在“对象引用被修改”时，插入一段钩子函数，记录修改记录，避免出现浮动垃圾无法被识别的问题—— 这也是G1浮动垃圾比CMS少的原因。

##### （3）最终标记（STW，快速）

相当于CMS的“重新标记”，但效率更高：利用并发标记阶段记录的“引用修改记录”，快速修正标记结果，清除浮动垃圾，标记出所有真正的存活对象；同时，统计每个Region中的“垃圾占比”（比如Region A垃圾占比90%，Region B垃圾占比30%），为后续“按需回收”做准备。

特点：STW时间比CMS的重新标记更短，因为不需要重新遍历整个引用链，只需修正变化的部分。

##### （4）筛选回收（STW，可控制时间）

这是G1的核心阶段，也是唯一可以控制STW时间的阶段，核心逻辑：**根据每个Region的垃圾占比，优先回收垃圾占比高的Region，同时兼顾“最大STW时间”参数**。

具体流程：

1. G1会对所有Region排序，筛选出“垃圾占比高、回收收益大”的Region（比如垃圾占比>70%的Region）；
    
2. 根据 `-XX:MaxGCPauseMillis` 设置的最大STW时间，选择“回收这些Region所需时间 ≤ 最大STW时间”的Region集合；
    
3. 启动多个GC线程（并行回收），对选中的Region进行回收：新生代Region采用复制算法，老年代Region采用标记-整理算法；
    
4. 回收完成后，释放内存，同时整理存活对象，消除内存碎片。
    

实例：设置 `-XX:MaxGCPauseMillis=100`（最大STW时间100ms），G1筛选出10个垃圾占比高的Region，回收前5个Region需要80ms（≤100ms），就只回收这5个Region，剩余5个Region等到下一次GC再回收，确保STW时间不超标。

#### 4. G1的适用场景和启动参数

##### 适用场景（重点）：

- 堆内存较大的应用（≥4GB，推荐8GB以上），比如Java后端服务、微服务、电商平台；
    
- 对延迟有明确要求（比如STW时间≤100ms），同时需要兼顾吞吐量的场景，比如电商支付、秒杀系统、接口服务；
    
- 替代CMS回收器，解决CMS的内存碎片、并发失败等问题。
    

##### 核心启动参数（必记）：

- -XX:+UseG1GC：强制使用G1回收器（JDK9及以上默认使用，JDK8需要手动开启）；
    
- -XX:MaxGCPauseMillis=100：设置最大STW时间（默认200ms），根据业务需求调整；
    
- -XX:G1HeapRegionSize=16m：设置每个Region的大小（可选，默认自动计算）；
    
- -XX:G1NewSizePercent=5：设置新生代最小比例（默认5%），避免新生代过小导致Minor GC频繁；
    
- -XX:G1MaxNewSizePercent=60：设置新生代最大比例（默认60%），避免新生代过大占用老年代内存。

为了方便你复习和选型，整理了常用5种回收器（Serial、Parallel、CMS、G1）的核心对比，结合适用场景和核心优缺点，清晰区分，避免混淆：

## 六、GC调优实战（重点，解决实际问题）

学习GC的最终目的，是解决实际开发中的“GC异常”问题——比如Full GC频繁、STW时间过长、内存溢出等，这部分结合你可能遇到的实际场景，讲解核心调优思路、步骤和实例，让你能直接落地使用。

核心调优原则：**先定位问题，再优化参数，最后验证效果**—— 不要盲目调参，否则不仅无法解决问题，还可能导致GC更异常；所有调优都要基于“GC日志”和“监控数据”，精准定位问题根源。

### 6.1 第一步：定位GC问题（3种核心方式）

要调优，首先要知道“问题出在哪里”—— 是Full GC频繁？还是STW时间太长？还是内存溢出？主要通过以下3种方式定位，结合使用效果最佳。

#### （1）打印GC详细日志（最基础、最常用）

通过JVM参数打印GC日志，分析日志中的“GC触发频率、回收内存大小、STW时间”，定位问题根源。

常用日志参数（必记，直接复制使用）：

```java
// 基础日志：打印GC时间、回收大小、STW时间
-XX:+PrintGCDateStamps  // 打印GC触发的时间戳（便于定位具体时间的GC异常）
-XX:+PrintGCDetails     // 打印GC详细信息（回收区域、回收前后内存、STW时间）
-Xloggc:./gc.log        // 将GC日志输出到指定文件（./gc.log表示当前目录）
-XX:+HeapDumpOnOutOfMemoryError  // 内存溢出时，自动生成堆转储文件（分析内存溢出原因）
-XX:HeapDumpPath=./heapdump.hprof  // 堆转储文件输出路径
```

日志分析重点（3个核心）：

- Full GC频率：如果1分钟触发多次Full GC，说明存在问题（比如内存碎片多、对象创建过快）；
    
- STW时间：如果单次STW时间超过100ms，且频繁出现，会影响应用响应；
    
- 回收效率：如果每次GC回收的内存很少（比如每次Full GC只回收10%的内存），说明大部分对象都是存活对象，可能存在内存泄漏。
    

实例日志片段（简化版）及分析：

```text
2025-12-07T22:00:00.123+0800: [Full GC (Allocation Failure) [PSYoungGen: 512M->0M(1024M)] [ParOldGen: 4096M->3800M(4096M)] 4608M->3800M(5120M), [Metaspace: 256M->256M(1024M)], 0.8900000 secs] [Times: user=3.56 sys=0.00, real=0.89 secs]
```

分析：

- 触发Full GC的原因是“Allocation Failure”（内存分配失败）；
    
- 老年代（ParOldGen）回收前4096M（满了），回收后3800M，只回收了296M，回收效率极低；
    
- STW时间（real=0.89 secs）= 890ms，超过100ms，会导致应用卡顿；
    
- 问题根源：老年代内存不足，且存活对象多，可能存在内存泄漏，或老年代设置过小。
    

#### （2）使用监控工具查看GC状态（可视化，更直观）

对于生产环境或本地调试，使用监控工具可以更直观地查看GC状态，无需手动分析日志，推荐3个常用工具（适合学生和开发者）：

- JVisualVM（JDK自带，无需额外安装）：
    
- 打开方式：JDK安装目录/bin/jvisualvm.exe，启动后连接本地Java进程，在“监控”面板查看GC情况（新生代、老年代内存变化、GC触发频率），在“抽样器”面板查看对象创建情况，定位内存泄漏；
    
- 核心优势：免费、易用，适合本地调试和小型应用监控；
    
- MAT（Memory Analyzer Tool，内存分析神器）：
    
- 用途：专门分析“堆转储文件（.hprof）”，定位内存泄漏—— 当应用发生内存溢出时，通过 `-XX:+HeapDumpOnOutOfMemoryError` 生成堆转储文件，用MAT打开，可快速找到“占用内存最多的对象”“导致内存泄漏的引用链”；
    
- 下载地址：https://www.eclipse.org/mat/（免费，支持Windows/Mac）；
    
- Arthas（阿里开源，生产环境首选）：
    
- 用途：在线排查Java应用问题，无需重启应用，可查看GC状态、对象创建、线程状态等；
    
- 核心命令：`jstat -gc 进程ID 1000`（每隔1秒打印一次GC状态）、`heapdump`（生成堆转储文件）；
    
- 安装方式：官网https://arthas.aliyun.com/，按照文档执行命令即可安装，适合后端服务调试。
    

#### （3）排查内存泄漏（最常见的GC问题根源）

很多时候，GC异常（Full GC频繁、内存溢出）的根源是“内存泄漏”—— 没用的对象被错误引用，无法被GC回收，久而久之占用大量内存，导致GC频繁触发。

常见的内存泄漏场景（结合Java代码实例，你更容易理解）：

- 场景1：静态集合类引用大量对象（最常见）
    

```java
public class MemoryLeakTest {
    // 静态List，和类绑定，生命周期和程序一致
    private static List<User> userList = new ArrayList<>();

    public static void main(String[] args) {
        // 循环创建10000个User对象，添加到静态List中
        for (int i = 0; i < 10000; i++) {
            User user = new User();
            userList.add(user);
        }
        // 后续不再使用userList，但没有释放引用，所有User对象无法被回收
        // 解决方案：使用完后，执行 userList.clear() + userList = null;
    }
}
```

原因：静态集合类（userList）和类绑定，只要类不被卸载，集合中的对象就会一直被引用，无法被GC回收，导致内存泄漏。

- 场景2：关闭资源未释放（IO流、数据库连接、Socket）
    

```java
public void readFile() {
    FileInputStream fis = null;
    try {
        fis = new FileInputStream("test.txt");
        // 读取文件内容，未关闭流
    } catch (IOException e) {
        e.printStackTrace();
    }
    // 未执行finally关闭流，fis引用的对象无法被回收
    // 解决方案：在finally中执行 fis.close();
}
```

原因：IO流、数据库连接等资源，底层依赖native方法，JVM无法自动回收，必须手动关闭（close()），否则会导致资源泄漏，同时占用内存。

- 场景3：匿名内部类/非静态内部类引用外部类
    

```java
public class OuterClass {
    public InnerClass getInnerClass() {
        // 匿名内部类，默认引用外部类OuterClass的实例
        return new InnerClass() {};
    }

    // 非静态内部类
    class InnerClass {}

    public static void main(String[] args) {
        OuterClass outer = new OuterClass();
        InnerClass inner = outer.getInnerClass();
        // 释放外部类引用，但内部类还引用着外部类，导致外部类无法被回收
        outer = null;
    }
}
```

原因：非静态内部类、匿名内部类，会默认持有外部类的实例引用，即使外部类的引用被释放，内部类依然引用着外部类，导致外部类无法被GC回收，造成内存泄漏。

内存泄漏排查步骤（结合工具）：

1. 通过GC日志，发现“Full GC频繁、回收效率低”；
    
2. 生成堆转储文件（.hprof），用MAT打开；
    
3. 在MAT中查看“Top Consumers”（占用内存最多的对象）；
    
4. 查看该对象的“引用链”，找到“不该存在的引用”（比如静态集合引用、未关闭的资源）；
    
5. 修改代码，释放无用引用，解决内存泄漏。
    

### 6.2 第二步：核心调优参数（按回收器分类，直接复用）

调优参数无需死记硬背，根据“回收器类型”和“问题场景”选择对应的参数，以下是常用参数（按回收器分类，结合调优场景，方便你直接复制使用）。

#### （1）通用参数（所有回收器都适用）

- -Xms：初始堆内存（推荐设置为和-Xmx一致，避免堆内存动态调整，减少GC）；
    
- 示例：-Xms10g -Xmx10g（初始和最大堆内存都是10G，固定堆内存）；
    
- -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:./gc.log：打印GC详细日志；
    
- -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=./heapdump.hprof：内存溢出时生成堆转储文件；
    
- -XX:MaxTenuringThreshold=15：设置对象晋升老年代的年龄阈值（默认15，新生代对象经历15次Minor GC晋升老年代）。
    

#### （2）Parallel回收器（吞吐量优先）调优参数

适用场景：后端服务、数据处理，追求高吞吐量，对延迟不敏感。

- -XX:+UseParallelOldGC：强制使用Parallel Scavenge + Parallel Old组合（推荐）；
    
- -XX:ParallelGCThreads=8：设置GC线程数量（推荐等于CPU核心数，比如8核CPU设置为8）；
    
- -XX:MaxGCPauseMillis=200：设置最大STW时间（尽量控制在200ms以内）；
    
- -XX:GCTimeRatio=99：设置GC时间占比（99表示GC时间不超过总时间的1%，吞吐量不低于99%）；
    
- -XX:+UseAdaptiveSizePolicy：开启自适应调参（JVM自动优化堆内存、Survivor区比例等，无需手动调参）。
    

示例调优参数（8核CPU，10G堆内存，后端服务）：

```text
-Xms10g -Xmx10g -XX:+UseParallelOldGC -XX:ParallelGCThreads=8 -XX:MaxGCPauseMillis=200 -XX:GCTimeRatio=99 -XX:+UseAdaptiveSizePolicy -XX:+PrintGCDetails -Xloggc:./gc.log
```

#### （3）CMS回收器（延迟优先）调优参数

适用场景：JDK8及以下，延迟敏感场景（如电商支付），但需解决其固有缺陷。

- -XX:+UseConcMarkSweepGC：强制使用ParNew + CMS组合；
    
- -XX:+UseParNewGC：新生代使用ParNew回收器（并行回收，加快新生代回收速度）；
    
- -XX:ParallelCMSThreads=4：设置CMS的GC线程数量（推荐为CPU核心数的1/2，避免占用过多CPU）；
    
- -XX:CMSInitiatingOccupancyFraction=80：设置CMS触发阈值（80%，更早触发回收，避免并发失败）；
    
- -XX:+UseCMSCompactAtFullCollection：Full GC后执行内存整理，减少内存碎片；
    
- -XX:CMSFullGCsBeforeCompaction=5：设置多少次Full GC后执行一次内存整理（默认5次，可调整为3次）；
    
- -XX:+CMSClassUnloadingEnabled：允许CMS回收元空间的类（避免元空间满触发Full GC）。
    

示例调优参数（8核CPU，10G堆内存，延迟敏感场景）：

```text
-Xms10g -Xmx10g -XX:+UseConcMarkSweepGC -XX:+UseParNewGC -XX:ParallelCMSThreads=4 -XX:CMSInitiatingOccupancyFraction=80 -XX:+UseCMSCompactAtFullCollection -XX:CMSFullGCsBeforeCompaction=3 -XX:+PrintGCDetails -Xloggc:./gc.log
```

#### （4）G1回收器（兼顾延迟和吞吐量）调优参数

适用场景：JDK8及以上，大内存堆（≥4GB），兼顾延迟和吞吐量（推荐首选）。

- -XX:+UseG1GC：强制使用G1回收器（JDK9及以上默认）；
    
- -XX:MaxGCPauseMillis=100：设置最大STW时间（核心参数，根据业务需求调整，比如延迟敏感设置为100ms）；
    
- -XX:G1HeapRegionSize=16m：设置每个Region的大小（10G堆内存，16m Region，共625个Region）；
    
- -XX:ParallelGCThreads=8：设置GC线程数量（等于CPU核心数）；
    
- -XX:ConcGCThreads=4：设置并发GC线程数量（推荐为CPU核心数的1/2）；
    
- -XX:G1NewSizePercent=10：设置新生代最小比例（10%，避免新生代过小导致Minor GC频繁）；
    
- -XX:G1MaxNewSizePercent=60：设置新生代最大比例（60%，避免新生代过大占用老年代）。
    

示例调优参数（8核CPU，16G堆内存，电商服务，延迟敏感）：

```text
-Xms16g -Xmx16g -XX:+UseG1GC -XX:MaxGCPauseMillis=100 -XX:G1HeapRegionSize=16m -XX:ParallelGCThreads=8 -XX:ConcGCThreads=4 -XX:G1NewSizePercent=10 -XX:G1MaxNewSizePercent=60 -XX:+PrintGCDetails -Xloggc:./gc.log
```

### 6.3 第三步：常见GC问题解决方案

结合你可能遇到的4种常见GC问题，给出具体的解决方案，结合调优参数和代码修改，让你能直接解决实际问题。

#### 问题1：Full GC频繁（每分钟多次）

核心原因（按概率排序）：1. 内存碎片过多（CMS回收器）；2. 老年代内存不足；3. 内存泄漏；4. 大对象频繁进入老年代。

解决方案：

1. 如果使用CMS回收器：切换为G1回收器（彻底解决内存碎片问题），或开启CMS内存整理参数（-XX:+UseCMSCompactAtFullCollection）；
    
2. 增大老年代内存：通过-Xmx增大堆内存，同时调整老年代比例（比如将堆内存从10G调整为16G）；
    
3. 排查内存泄漏：生成堆转储文件，用MAT分析，找到占用内存最多的对象，修改代码释放无用引用；
    
4. 调整对象晋升阈值：如果大对象频繁进入老年代，可通过-XX:MaxTenuringThreshold调整晋升年龄，让对象在新生代多停留一段时间，被Minor GC回收；
    
5. 避免大对象直接进入老年代：合理设置新生代内存（增大Eden区），让大对象能在新生代被回收。
    

#### 问题2：STW时间过长（超过1秒）

核心原因：1. Full GC频繁；2. GC线程数量过少；3. 堆内存过大，回收耗时久；4. 存活对象过多，复制/整理开销大。

解决方案：

1. 减少Full GC频率：参考问题1的解决方案，避免Full GC频繁触发；
    
2. 调整GC线程数量：根据CPU核心数，设置合适的GC线程数量（ParallelGCThreads=CPU核心数，ConcGCThreads=CPU核心数/2）；
    
3. 使用G1回收器：设置-XX:MaxGCPauseMillis，控制STW时间，让G1按需回收，避免回收整个堆内存；
    
4. 拆分堆内存：如果堆内存过大（比如32G），可拆分多个JVM实例，减少单个堆内存大小，缩短回收时间；
    
5. 减少存活对象数量：排查内存泄漏，释放无用引用，减少每次GC需要处理的存活对象。
    

#### 问题3：内存溢出（OutOfMemoryError）

核心原因（3种常见）：1. 内存泄漏（最常见）；2. 堆内存设置过小，无法满足应用需求；3. 大对象过多，内存无法容纳。

解决方案（按优先级）：

1. 排查内存泄漏：生成堆转储文件，用MAT分析引用链，修改代码释放无用引用（比如静态集合clear()、关闭IO流）；
    
2. 增大堆内存：通过-Xms和-Xmx增大堆内存，比如从5G调整为10G，根据应用实际内存占用调整；
    
3. 处理大对象：避免创建过大的对象（比如一次性读取超大文件到内存），可分批次处理（比如分批次读取文件，处理完一批释放一批）；
    
4. 调整回收器：使用G1回收器，其Humongous Region专门存放大对象，减少大对象对内存的影响。
    

#### 问题4：Minor GC频繁（每秒多次）

核心原因：1. 新生代内存过小，Eden区很快被占满；2. 临时对象创建过快，Eden区回收速度赶不上对象创建速度。

解决方案：

1. 增大新生代内存：调整堆内存中新生代的比例（比如G1设置-XX:G1NewSizePercent=20，增大新生代最小比例）；
    
2. 增大Eden区：通过-XX:SurvivorRatio=8（Eden区和Survivor区比例8:1），增大Eden区大小，减少Minor GC触发频率；
    
3. 优化代码：减少临时对象的创建（比如循环中避免频繁创建对象，可复用对象）；
    
4. 调整对象晋升阈值：适当提高-XX:MaxTenuringThreshold，让临时对象在新生代被回收，避免过早晋升到老年代。
    

## 七、总结

本文围绕Java垃圾回收机制，从“核心概念→垃圾判断→回收算法→分代回收→回收器→调优实战”，层层递进，覆盖所有核心知识点，结合代码实例和通俗解释，适合你作为学习笔记沉淀，以下是核心知识点梳理（重点记忆）：

### 1. 核心基础

- GC本质：JVM的自动内存回收机制，解决内存泄漏和内存溢出问题，无需手动释放内存；
    
- GC触发场景：堆内存不足（最常见）、手动调用System.gc()、JVM参数配置、内存使用率达阈值；
    
- 垃圾判断：主流是“可达性分析算法”，通过GC Roots追溯引用链，不可达对象为垃圾；GC Roots有4类（虚拟机栈局部变量、本地方法栈引用、方法区静态属性、方法区常量）。
    

### 2. 核心算法（3种）

- 标记-清除：基础算法，有内存碎片，适用于老年代；
    
- 标记-整理：解决碎片问题，有对象移动开销，适用于老年代；
    
- 复制算法：无碎片、效率高，内存浪费一半，适用于新生代。
    

### 3. 分代回收机制（JVM核心设计）

- 堆内存划分：新生代（1/3）+ 老年代（2/3）；新生代=Eden区（80%）+ 2个Survivor区（各10%）；
    
- 新生代回收（Minor GC）：复制算法，触发频繁，STW时间短；
    
- 老年代回收（Major GC）：标记-清除/标记-整理，触发频率低，STW时间长；
    
- Full GC：回收新生代+老年代，STW时间最长，尽量避免。
    

### 4. 垃圾回收器（实战选型）

- Serial：单线程，客户端适用；
    
- Parallel：并行，吞吐量优先，后端服务适用；
    
- CMS：并发，延迟优先，JDK8及以下延迟敏感场景适用（有缺陷）；
    
- G1：区域化，兼顾延迟和吞吐量，大内存堆、主流场景首选（JDK9+默认）。
    

### 5. GC调优核心

- 调优原则：先定位问题（GC日志+监控工具），再调参，最后验证；
    
- 核心目标：减少Full GC频率、控制STW时间、避免内存溢出；
    
- 常见问题：Full GC频繁、STW过长、内存溢出、Minor GC频繁，对应解决方案落地即可。
